# Концепция: Сервис мониторинга и сравнения цен на запчасти

## 1. Идея и цель

Сервис автоматически собирает данные о запчастях (мототехника/спецтехника AODES и
аналогичные дистрибьюторы) с сайтов поставщиков, нормализует карточки товаров и
сравнивает цены на одинаковые позиции между источниками. Результат — отчёты и
дашборд: где дешевле купить конкретную деталь, динамика цен, расхождения по
регионам/дистрибьюторам.

**Стартовые источники:**
- https://aodesf7.ru/zapchasti/ — дистрибьютор AODES, ЦФО/ПФО/ЮФО/СКФО
- https://aodes-mdv.ru/zapchasti/ — дистрибьютор AODES, другой регион

Оба сайта по факту продают одного и того же производителя (AODES), поэтому
артикулы/OEM-номера деталей должны частично пересекаться — это ключевое для matching.

> Прямой технический аудит вёрстки этих сайтов из текущего окружения недоступен
> (сетевая политика блокирует исходящий доступ к этим доменам). Архитектура ниже
> рассчитана на разведку селекторов как первый шаг реализации, без жёсткой
> привязки к конкретной структуре HTML.

## 2. Пользователи и сценарии

- **Закупщик/менеджер**: хочет видеть таблицу "деталь → цена у каждого
  поставщика → где дешевле" и получать алерт при изменении цены.
- **Аналитик**: хочет историю цен и экспорт в Excel/Google Sheets.
- **Будущее расширение**: добавление новых сайтов-источников без переписывания
  ядра системы.

## 3. Архитектура (высокий уровень)

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Site Adapter │ ... │  Site Adapter │     (по одному на сайт-источник)
│   aodesf7     │     │  aodes-mdv    │
└──────┬───────┘     └──────┬───────┘
       │ raw items (JSON)    │
       ▼                     ▼
┌─────────────────────────────────────┐
│        Normalizer / Cleaner          │  → единая схема товара
└──────────────┬────────────────────────┘
               ▼
┌─────────────────────────────────────┐
│     Matching Engine (артикул→fuzzy)  │  → group_id для одинаковых деталей
└──────────────┬────────────────────────┘
               ▼
┌─────────────────────────────────────┐
│   Storage (Postgres): products,      │
│   price_history, sources, matches    │
└──────────────┬────────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  Analytics / API / Dashboard         │  → сравнение цен, алерты, экспорт
└─────────────────────────────────────┘

         ▲
         │ orchestrates run schedule, retries, anti-bot fallback
┌─────────────────────────────────────┐
│           Scheduler / Orchestrator   │
└─────────────────────────────────────┘
```

### 3.1 Site Adapter (сбор данных)

Гибридный подход, два уровня одного адаптера на сайт:

1. **HTTP-уровень** (requests + BeautifulSoup/lxml, либо HTTP-клиент с заголовками
   браузера) — пробуем первым, дешевле и быстрее.
2. **Headless-уровень** (Playwright) — включается автоматически, если:
   - HTTP-ответ — 403/429/капча/пустой DOM без товаров;
   - в каталоге обнаружен JS-рендеринг (пагинация через XHR/SPA-каркас).

Каждый адаптер реализует общий интерфейс:

```
class SiteAdapter:
    def fetch_categories(self) -> list[Category]
    def fetch_items(self, category) -> Iterator[RawItem]
```

`RawItem` — минимально: `name, sku/article, price, currency, availability,
category_path, image_url, source_url, scraped_at`.

Конфиг селекторов (CSS/XPath) для каждого сайта хранится отдельно от кода
(YAML/JSON), чтобы при изменении вёрстки править конфиг, а не логику.

### 3.2 Anti-bot / устойчивость

- Ротация User-Agent, разумные задержки между запросами, лимит параллелизма
  на домен (вежливый краулинг, не DDoS).
- Уважение `robots.txt`; сбор только публично доступных страниц каталога.
- Retry с экспоненциальным backoff на сетевые ошибки/5xx.
- Кэш последнего успешного снимка по сайту — если краулинг сайта упал, не
  затирать историю нулями, а помечать `source_status=stale`.
- Headless-режим — fallback, а не основной путь, чтобы не перегружать сайты.

### 3.3 Normalizer

Приводит разнородные карточки к единой схеме `Product`:
- очистка цены (валюта, пробелы, "руб."), приведение к числу;
- извлечение артикула/OEM-номера из текста, если он не в отдельном поле;
- нормализация наименования (нижний регистр, удаление брендовых стоп-слов,
  единиц измерения) — нужно и для хранения, и как вход для fuzzy-matching.

### 3.4 Matching Engine

Комбинированная стратегия (как и обсуждали):

1. **Точное совпадение по артикулу/OEM-номеру** — основной путь. Нормализуем
   артикул (убираем пробелы/дефисы/регистр) и матчим 1:1.
2. **Fallback на fuzzy-matching по названию**, если у одной из карточек нет
   артикула или артикулы не совпали:
   - токенизация названия + модели техники;
   - similarity (например, RapidFuzz/token_sort_ratio) с порогом;
   - кандидаты с similarity выше порога, но не 100%, уходят в очередь
     **ручной проверки** (модерация), а не автоматически считаются совпадением —
     цена это деньги, ложный матч хуже отсутствия матча.
3. Результат matching — `match_group_id`, объединяющий товары из разных
   источников как "одна и та же деталь".

### 3.5 Storage

Postgres, примерная схема:
- `sources` (id, name, base_url, status)
- `products` (id, source_id, sku, name, raw_name, category, image_url, source_url)
- `price_history` (product_id, price, currency, availability, scraped_at)
- `match_groups` (id, canonical_name, canonical_sku)
- `product_matches` (product_id, match_group_id, match_method, confidence)
- `match_review_queue` (для fuzzy-кандидатов, ожидающих подтверждения)

### 3.6 Analytics / API / Dashboard

- API (REST) для получения: список групп товаров, цены по источникам, история,
  дельта цены, "где дешевле".
- Дашборд (минимально — таблица с фильтрами + графики истории цены).
- Алерты (email/Telegram) при появлении новой минимальной цены или скачке.

### 3.7 Scheduler

Cron-подобный планировщик (например, Celery beat / APScheduler) запускает
сбор по каждому источнику с независимым расписанием (чтобы падение одного
сайта не блокировало остальные), плюс ручной запуск "собрать сейчас".

## 4. Стратегия разработки (этапы)

### Этап 0 — Разведка (1 спринт)
- Технический аудит вёрстки `aodesf7.ru` и `aodes-mdv.ru`: структура каталога,
  наличие артикулов в карточках, есть ли API/sitemap, пагинация, антибот-защита.
- Прототип: вручную выгрузить 1 категорию с каждого сайта, проверить, реально
  ли сопоставить позиции по артикулу.
- Решение go/no-go по объёму fuzzy-matching, нужному на старте.

### Этап 1 — MVP сбора данных
- Реализовать `SiteAdapter` для двух сайтов (HTTP-уровень + конфиг селекторов).
- Normalizer + схема БД + сохранение снимков цен.
- CLI-команда "собрать всё сейчас" без UI.

### Этап 2 — Matching и сравнение цен
- Matching Engine (артикул + fuzzy fallback + очередь ручной проверки).
- Простой отчёт/API: список совпавших товаров с ценами по источникам.

### Этап 3 — Автоматизация и устойчивость
- Scheduler с независимыми расписаниями по источникам.
- Headless fallback для сайтов с защитой/JS-рендерингом.
- Мониторинг сбоев краулинга (алерт, если источник не собрался N раз).

### Этап 4 — Дашборд и алерты
- Веб-интерфейс: таблица сравнения, история цены, фильтры по категориям.
- Алерты на изменение цены/появление новой минимальной цены.

### Этап 5 — Масштабирование на новые источники
- Onboarding нового сайта = новый адаптер + конфиг селекторов, без изменения
  ядра matching/storage/API.
- (Опционально) полуавтоматический помощник для разметки селекторов нового сайта.

## 5. Стек (предложение)

- **Сбор данных**: Python, `requests`/`httpx` + `BeautifulSoup`/`lxml`,
  fallback `Playwright`.
- **Matching**: `rapidfuzz` для fuzzy-сравнения.
- **Backend/API**: Python (FastAPI) или Node.js (NestJS) — на выбор команды.
- **БД**: PostgreSQL.
- **Очереди/планировщик**: Celery + Redis, либо APScheduler для старта без
  лишней инфраструктуры.
- **Дашборд**: минимально — серверный рендер таблиц; позже — React/Vue SPA.

## 6. Риски

| Риск | Митигация |
|---|---|
| Сайты меняют вёрстку → парсер ломается | Селекторы в конфиге, не в коде; алерты при падении парсера до нуля товаров |
| Антибот-блокировка / юридические ограничения на скрапинг | Вежливый краулинг, соблюдение robots.txt, только публичные данные, throttling |
| Отсутствие артикула на части карточек | Fuzzy fallback + очередь ручной модерации, не автоматическое доверие низкой confidence |
| Ложные совпадения "одинаковых" деталей | Порог confidence + ручная проверка для среднего диапазона similarity |
| Рост числа источников увеличивает нагрузку/расходы на хостинг headless-браузеров | HTTP-уровень как основной путь, headless только как fallback |

## 7. Метрики успеха MVP

- Доля товаров с успешным matching по артикулу ≥ ожидаемого порога (определить после Этапа 0).
- Доля карточек, ушедших в fuzzy-очередь ручной проверки.
- Стабильность сбора: % успешных запусков краулера за неделю без падений.
- Время от изменения цены на сайте-источнике до появления в дашборде.
